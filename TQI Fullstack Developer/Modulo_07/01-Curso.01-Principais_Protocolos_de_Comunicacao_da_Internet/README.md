<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m03s956.jpg" alt="" width="840">
</p>

O HTTP significa HyperText Transfer Protocol. Ele atua como o protocolo de comunicação  responsável por intermediar a troca de informações entre duas pontas. O navegador implementa o lado do Cliente HTTP , que solicita recursos a um Servidor , este sendo responsável por hospedar os objetos web. Historicamente, as diretrizes desse protocolo foram estabelecidas pelas RFCs 1945 e 2116.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m08s012.jpg" alt="" width="840">
</p>

Dinâmica inicial da comunicação. O Cliente envia ativamente mensagens de requisição (Request HTTP)  pedindo os recursos desejados ao Servidor. Para que essas mensagens trafeguem com segurança e com controle de fluxo, o HTTP apoia-se no Protocolo TCP  da camada de transporte subjacente.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m11s803.jpg" alt="" width="840">
</p>

Após processar a requisição (Request), o Servidor retorna os objetos web pertinentes por meio de mensagens de resposta (Response HTTP). Esse fluxo interdependente de papéis caracteriza uma Arquitetura Client-server clássica.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m14s659.jpg" alt="" width="840">
</p>

Uma característica estrutural crítica do Servidor no protocolo HTTP é ser "Stateless" (sem estado). Isso denota que o servidor atende às requisições de forma independente e isolada, não guardando o estado do cliente ou rastreando repetições entre diferentes chamadas.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m18s567.jpg" alt="" width="840">
</p>

Linha do tempo resumida demonstrando a evolução do protocolo HTTP em resposta às mudanças na web ao longo das décadas. A tecnologia iniciou com o HTTP 0.9 em 1991 , seguido pelo HTTP 1.0 em 1996. O HTTP 1.1, lançado em 1997 , inseriu funcionalidades críticas ainda muito utilizadas, como o CORS e o Keep-Alive. Mais tarde, inovações com o SPDY em 2012 prepararam o terreno para a homologação do HTTP 2.0 em 2015.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m23s200.jpg" alt="" width="840">
</p>

Exemplo prático da estrutura textual que compõe os dois tipos fundamentais de mensagens HTTP: Request (requisição) e Response (resposta). É possível identificar o uso do método `Get`, diretrizes de host e linguagens na requisição, bem como o cabeçalho de resposta emitido por um servidor do tipo Apache, atestando sucesso (`200 OK`).

```http
Get: /somedir/page.html HTTP/1.1
Host: www.someschool.edu
Connection:close
User-agent: Mozilla/5.0
Accpet-language:fr

HTTP/1.1 200 OK
Connection: close
Date: Tue, 09 Aug 2011 15:44:04 GMT
Server: Apache/2.2.3 (CentOS)
Last-Modified: Tue, 09 Aug 2011 15:11:03 GMT
Content-Length: 6821
Content-Type: text/html
(data, data, ....)
```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m27s023.jpg" alt="" width="840">
</p>

Detalhamento visual do envio de uma requisição utilizando o método GET. Este método é construído pelo cliente com a intenção estrita de capturar um recurso do lado do servidor. O navegador (User-agent) formata uma solicitação Request HTTP apontando, por exemplo, para a página inicial da Amazon.

```http
Get: https://www.amazon.com.br
Host: amazon.com
Connection:close
User-agent: Mozilla/5.0
```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m29s307.jpg" alt="" width="840">
</p>

Ação de busca do lado do servidor após o recebimento da solicitação GET. O servidor da aplicação irá consultar sua estrutura interna de arquivos, diretórios e bancos de dados lógicos para encontrar o recurso ou as informações requisitadas pelo cliente.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m30s719.jpg" alt="" width="840">
</p>

Conclusão do ciclo de uma requisição GET. Ao resgatar e preparar os dados internamente, o servidor gera uma mensagem do tipo HTTP Response e a entrega de volta ao cliente informando sucesso (`HTTP/1.1 200 OK`), contendo também a representação do conteúdo demandado (como a estrutura HTML da página solicitada).

```http
HTTP/1.1 200 OK
Connection: close
Date: Tue, 09 Aug 2011 15:44:04 GMT
Server: Apache/2.2.3 (CentOS)
Last-Modified: Tue, 09 Aug 2011 15:11:03 GMT
Content-Length: 6821
Content-Type: text/html

(data ... Amazon initial page)
```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m32s337.jpg" alt="" width="840">
</p>

Apresentação da dinâmica do método POST, cujo papel principal é enviar dados elaborados do cliente ao servidor. Este fluxo é classicamente empregado ao submeter formulários de cadastro, repassando no corpo da mensagem um conjunto robusto de dados como nome e contato.

```http
Post: https://www.amazon.com.br/cadastro
Host: amazon.com
Connection:close
User-agent: Mozilla/5.0

...
Data: nome, data de nascimento, contato
...
```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m36s622.jpg" alt="" width="840">
</p>

Ação do servidor após recepção de uma submissão POST. O servidor processa as informações e executa as ações devidas de gravação e atualização de registro no Sistema Gerenciador de Banco de Dados (SGBD).

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m38s062.jpg" alt="" width="840">
</p>

Conclusão da operação via POST. Tendo sido as informações corretamente manipuladas no banco de dados, o servidor responde com um sinalizador de sucesso (`200 OK`), confirmando ao cliente que a ação de atualizar/inserir dados foi efetivada.

```http
HTTP/1.1 200 OK
Connection: close
Date: Tue, 09 Aug 2011 15:44:04 GMT
Server: Apache/2.2.3 (CentOS)
Last-Modified: Tue, 09 Aug 2011 15:11:03 GMT
Content-Length: 6821
Content-Type: text/html

(data ... Update Succeed)
```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m43s153.jpg" alt="" width="840">
</p>

Visão panorâmica de como os dados são organizados no corpo (Entity Body) da mensagem HTTP, sejam eles despachados ou requisitados. A finalidade dessa organização envolve três etapas centrais: o Armazenamento, a Transmissão segura entre os nós, e a Reconstrução dos dados recebidos do outro lado. Os formatos dominantes da indústria para realizar esse trabalho de padronização estrutural são o XML e o JSON.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m44s904.jpg" alt="" width="840">
</p>

Como declarar o tipo estrutural utilizado no HTTP Entity Body. O cliente e o servidor chegam a um consenso técnico através dos atributos declarados no cabeçalho da requisição (Request Header). Define-se o parâmetro `Content-Type` para sinalizar como a mensagem atual está formatada, bem como o atributo `Accept` indicando quais formatos a aplicação aceita receber de volta.

```http
Content-Type: application/json
Accept: application/json

```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m52s920.jpg" alt="" width="840">
</p>

Apresentação conceitual e visual do formato Extensible Markup Language (XML). Seu objetivo primário é realizar a "Serialization" (transformação de dados para intercâmbio) para viabilizar a Comunicação e embutir de forma robusta um conjunto denso de Metadados. Sintaticamente, assemelha-se ao HTML, estruturando os dados por meio de encapsulamento por tags.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<authentication-context>
    <username>my_username</username>
    <password>my_password</password>
    <validation-factors>
        <validation-factor>
            <name>remote_address</name>
            <value>127.0.0.1</value>
        </validation-factor>
    </validation-factors>
</authentication-context>
```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m54s206.jpg" alt="" width="840">
</p>

Apresentação do formato JavaScript Object Notation (JSON). Ele foi desenhado herdando a sintaxe da linguagem JavaScript (JS) com a premissa de ser altamente simples e um padrão "Lightweight" (leve). Apesar das raízes sintáticas, mantém-se totalmente independente de linguagem.

```json
{
    "username": "my_username",
    "password": "my_password",
    "validation-factors": {
        "validation-factor": [
            {
                "name": "remote_address",
                "value": "127.0.0.1"
            }
        ]
    }
}
```

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h03m57s548.jpg" alt="" width="840">
</p>

Pontos em comum entre o XML e o JSON. A despeito de suas diferenças, ambos provaram vasta utilização no mercado, pois possuem propriedades auto-descritivas e organizam a informação baseada em níveis hierárquicos rígidos. Ambos são formatos completamente independentes de linguagens de programação.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m00s294.jpg" alt="" width="840">
</p>

Análise das diferenças estruturais entre XML e JSON. A natureza do XML baseia-se maciçamente no uso de Tags para a delimitação verbal da informação. Por contrapartida, o JSON distingue-se por ser incrivelmente mais sucinto, priorizar extrema legibilidade via modelo dicionário chave-valor e harmonizar perfeitamente com a estruturação da informação através da utilização de arrays.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m02s247.jpg" alt="" width="840">
</p>

Critérios para optarmos entre XML ou JSON no design de uma aplicação. A escolha dita-se pelas variáveis do projeto: aplicações modernas que exigem tráfego veloz costumam pedir formatos para "Aplicações leves" (JSON). Em cenário oposto, arquiteturas de "Complexidade" maior que demandam extensa utilização de "Metadados", bem como dependências presas à "Tecnologia" legada, justificam e requerem o uso do XML.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m09s924.jpg" alt="" width="840">
</p>

Abertura de nova seção na aula destinada a investigar peculiaridades e deficiências técnicas inerentes especificamente ao Protocolo HTTP em sua versão 1.1.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m16s100.jpg" alt="" width="840">
</p>

Como o TCP (Transmission Control Protocol) atua abaixo do HTTP na camada de transporte. A base da conectividade é forjada a partir de um processo de aperto de mãos em três etapas de sincronização (`SYN`, `SYN ACK`, `ACK`). Essa conectividade fundamental estabelecida pode comportar-se tanto em caráter Persistente (conexão deixada aberta), quanto Não-persistente (conexão encerrada tão logo responda à solicitação).

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m21s647.jpg" alt="" width="840">
</p>

Dinâmica sequencial padrão do protocolo HTTP 1.1. O método base executa as requisições ordenadamente: exige-se que o cliente encaminhe primeiro uma requisição (ex: o código base em `main.html`), congele o estado até que a resposta total chegue e, apenas na sequência, engatilhe com sucesso uma segunda requisição necessária (ex: os complementos de estilo e lógica contidos em um `file.js`).

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m23s007.jpg" alt="" width="840">
</p>

Artifício implementado pela tecnologia de navegadores modernos (Browsers) no protocolo HTTP 1.1. Com o fito de contornar os graves problemas de lentidão oriundos do carregamento unicamente sequencial estrito, os desenvolvedores de browsers passaram a permitir canais paralelos para executar de forma simultânea e assíncrona até seis conexões de solicitações HTTP ativas com o mesmo servidor.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m25s347.jpg" alt="" width="840">
</p>

Permissão de conexão constante no HTTP 1.1. Se o tráfego exige permanecer conectado repetidamente sem a penalidade constante de reconstruir as saudações via rede para pacotes subsequentes, recorre-se a uma conexão de túnel TCP Persistente (Keep-Alive). Uma vez fixado um túnel principal, múltiplos sinais HTTP trafegam através desta avenida mantida operante.

<p align="center">
<img src="000-Midia_e_Anexos/vlcsnap-2026-02-23-15h04m28s059.jpg" alt="" width="840">
</p>

Sintese dos gargalos primários que maculam a arquitetura original do HTTP 1.1. Primeiramente, o encavalamento da fila se uma requisição anterior atrasa e entope o processamento subsequente, caracterizando o evento de bloqueio "HOL - Head of Line Blocking". Ademais, o formato sofre do inchaço estrutural e do peso da repetição ineficaz nos blocos de pacotes, pois toda vez um volume constante de dados textuais e cabeçalhos devem ser reenviados ao remetente, inundando e sufocando os conduítes da internet desnecessariamente.
