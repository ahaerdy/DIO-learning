O padrĂŁo arquitetural **REST** (Representational State Transfer) surgiu como uma alternativa mais simples e menos complexa ao protocolo SOAP. Enquanto o SOAP dependia de documentos XML extensos que exigiam traduĂ§Ăľes complexas de cĂłdigo , o REST utiliza a infraestrutura nativa do protocolo **HTTP** para realizar operaĂ§Ăľes de integraĂ§ĂŁo.

Neste ecossistema, o fluxo de comunicaĂ§ĂŁo baseia-se em:

* **Clients**: Dispositivos ou usuĂĄrios que enviam **HTTP Requests** e aguardam por uma resposta.
* **Rest API**: Recebe as requisiĂ§Ăľes, processa a lĂłgica necessĂĄria (como a criaĂ§ĂŁo de usuĂĄrios) e interage com o **Database**.
* **Verbos HTTP**: Definem a aĂ§ĂŁo a ser executada, como **GET** para leitura, **POST** para criaĂ§ĂŁo, **PUT** para atualizaĂ§ĂŁo total, **PATCH** para modificaĂ§ĂŁo parcial e **DELETE** para exclusĂŁo.
* **Endpoints**: Rotas especĂ­ficas como `/allUsers`, `/newUser` ou `/updateUser` que identificam o recurso alvo.
* **Response**: ApĂłs o processamento, a API retorna os dados ao cliente, tipicamente formatados em **JSON** ou **XML**.

| Uri             | Method  | DescriĂ§ĂŁo                                | Request            | Response          | Status Code |
|-----------------|---------|------------------------------------------|--------------------|-------------------|-------------|
| `/people`       | **GET** | Recupera todas as pessoas do sistema     | n/a                | Person collection | 200/404     |
| `/people/{id}`  | **GET** | ObtĂŠm uma pessoa especĂ­fica pelo ID      | n/a                | Person            | 200/404     |
| `/people`       | **POST**| Cria uma nova entidade no sistema        | Person (sem ID)    | Person            | 201/404     |
| `/people/{id}`  | **PUT** | Modifica ou substitui um recurso existente | Person             | n/a               | 200/404     |
| `/people/{id}`  | **DELETE** | Remove um recurso do sistema          | n/a                | n/a               | 200/404     |

O uso correto dos **Status Codes** (como o **201 Created** para POST ou **200 OK** para sucessos) permite que o cliente compreenda o resultado da operaĂ§ĂŁo sem ambiguidades.

Para medir o nĂ­vel de aderĂŞncia aos princĂ­pios REST, utiliza-se o **Modelo de Maturidade de Richardson**. Este modelo organiza a evoluĂ§ĂŁo de um serviĂ§o em quatro nĂ­veis distintos, conhecidos como "A GlĂłria do REST":

* **NĂ­vel 0 (The Swamp of POX)**: O uso do HTTP apenas como um meio de transporte para mensagens, sem explorar seus recursos semĂ˘nticos.

import com.fasterxml.jackson.databind.DeserializationFeature;
import com.fasterxml.jackson.databind.ObjectMapper;
import com.fasterxml.jackson.databind.SerializationFeature;
import com.fasterxml.jackson.datatype.jsr310.JavaTimeModule;

public class JsonConversionUtils {

    public static String asJsonString(Object beerDTO) {

            ObjectMapper objectMapper = new ObjectMapper();
            objectMapper.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
            objectMapper.configure(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS, false);
            objectMapper.registerModules(new JavaTimeModule());

        } catch (Exception e) {

        }
    }
